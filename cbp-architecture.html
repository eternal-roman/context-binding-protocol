Context Binding Protocol

Design Reference · experimental · MIT licensed

A small declarative format and reference library for delivering structured, state-conditioned, token-budgeted context to LLMs. It assembles well-established ideas — frame systems, typed property graphs, content-addressed identity, conditional rules, context compression — into one implementation, and claims no novelty over them.

I. Core Thesis

MCP answers: "How does the LLM call this tool?"
CBP answers: "What does the world look like right now, structured for reasoning?"

The protocol encodes contextual state — entities, their relationships, their relevance weights, and their decay properties — in a format designed for LLM consumption. Context is inherited downward, conditional on state, and compressible across conversation turns.

Scope and Applicability

CBP is a domain-general protocol. It applies to any business that has context-dependent communication with AI systems and needs a structured, state-aware, token-efficient way to bind that context. The cryptocurrency examples used throughout this document (BTC, ETH, market regimes) reflect the origin context in which CBP was first conceived; they are illustrative, not prescriptive. Section V enumerates six application domains — and the protocol is designed to extend to any business domain with the same structural needs. CBP is a horizontal protocol with a trading origin story, not a trading protocol.

II. Node Taxonomy

Every object in CBP is a context node. There are four node types, each serving a distinct function in the reasoning graph:

TypePurposeExamples
entity A thing that exists in the domain. The noun. Can hold state children. BTC, user:jon, ticker:AAPL, patient:4821
state A measurable property of an entity at a point in time. Always a child. price, momentum, sentiment, risk_score
prior A belief about an entity or state before evidence. The Bayesian input. regime:risk_on, sector:bullish, correlation_stable
frame A container that scopes a subset of the graph for injection. The viewport. frame:crypto_macro, frame:patient_intake

Node Schema (Canonical)

{
  "id":      "8-char hex",       // BLAKE3 of content fields (type, val, lineage, tags)
  "type":    "entity|state|prior|frame",
  "val":     <any JSON value>,   // payload — semantics vary by type
  "w":       0.0–1.0,           // relevance weight (metadata, not hashed)
  "decay":   "epoch|event|none", // when staleness triggers
  "ttl":     <seconds | null>,   // hard expiry
  "lineage": "parent_id | null", // inheritance chain; null for frame roots
  "tags":    ["domain:trading", "severity:high"],
  "v":       1,                  // optimistic concurrency counter
  "prev":    "prior_id | null"   // previous version of same logical node (G9)
}
Identity derivation. The id field is the first 8 hex characters of the BLAKE3 hash of the node's content fields — type, val, lineage, and tags. Metadata fields (w, decay, ttl, v, prev) are not hashed; changing a node's weight does not change its identity. On collision, the display id extends one character at a time. The formal schema is at spec/schemas/node.schema.json.
Note on supersedes versus prev (G9). The standard-8 edge vocabulary includes a supersedes edge type — a semantic relationship between two nodes expressing "A replaces B conceptually or temporally." This edge is observable by other graph participants and by conditional-edge evaluation. Separately, every node has an optional prev field — a pointer to the prior version of the same logical node, implementing the append-mostly history required by invariant #6. The two are not interchangeable: supersedes is a first-class graph relationship; prev is a node-internal version pointer. Implementations must not collapse them.

Inheritance Resolution

A child node inherits all fields from its parent except those it explicitly declares. Resolution walks upward through lineage until it hits a frame node (the root). This is prototypal — not class-based.

frame:crypto_macro w:1.0 decay:epoch tags:[domain:crypto] │ ├─ entity:BTC (inherits w:1.0, decay:epoch, tags) │ ├─ state:price val:68420 w:0.9 decay:event ← overrides │ ├─ state:momentum val:0.72 ← inherits w:1.0, decay:epoch │ └─ prior:regime val:risk_on w:0.6 ttl:3600 │ ├─ entity:ETH w:0.8 ← overrides, inherits decay, tags │ ├─ state:price val:3180 │ └─ state:gas val:12gwei decay:event tags:[+infra] │ └─ edge src:BTC tgt:ETH rel:correlates strength:0.85 conditional: regime=risk_on
Key insight: In this tree, state:momentum under BTC never declares w, decay, or tags — it inherits all three from the frame root through its entity parent. Serialized, those inherited fields are omitted, so the node is smaller than a fully specified one.

III. Edge Vocabulary

Edges are deliberately constrained. A small, fixed vocabulary means the LLM doesn't burn tokens parsing novel relationship semantics. Every edge the model encounters is one it's seen before.

Edge TypeSemanticsSymmetric?
causesA produces or triggers BNo
correlatesA and B move together (± strength)Yes
contradictsA's truth diminishes B's truthYes
qualifiesA modifies the interpretation of BNo
supersedesA replaces B (temporal override)No
requiresB is meaningless without ANo
inhibitsA's presence weakens B's relevance weightNo
amplifiesA's presence strengthens B's relevance weightNo

Conditional Edge Activation

{
  "src": "BTC",
  "tgt": "ETH",
  "rel": "correlates",
  "strength": 0.85,
  "conditional": {
    "field": "prior:regime.val",
    "op": "eq",
    "value": "risk_on"
  }
}

When the regime prior shifts to risk_off, this edge goes dormant. It's excluded from frame serialization. The LLM never sees it — and never wastes tokens reasoning about a relationship that doesn't apply in the current context.

Full Conditional Grammar

Conditionals support boolean composition. The complete grammar:

condition := "always"              // unconditional (syntactic sugar)
           | leaf                   // single comparison
           | { "all": [condition] } // AND — all must be true
           | { "any": [condition] } // OR — at least one must be true
           | { "not": condition }   // NOT — must be false

leaf := { "field": "<type>:<id>.<field>", "op": <op>, "value": <literal> }
      | { "field": "<type>:<id>.<field>", "op": "exists" }

op := "eq" | "ne" | "lt" | "lte" | "gt" | "gte"
    | "in" | "contains" | "matches" | "exists"

Type comparisons are strict — comparing a number to a string is an evaluation error, not false. The formal schema is at spec/schemas/edge.schema.json.

Why this matters: Conditional edges let the serialized graph reflect current state: a relationship can exist in storage but be excluded from a frame's serialization when its condition does not hold. This is ordinary conditional evaluation applied at the context-assembly layer.

IV. Serialization Tiers

The protocol's core memory trick. Three tiers of compression, negotiated between the CBP server and the consuming client:

TIER 1: FULL First encounter

Complete node/edge expansion. All fields explicit. Inheritance resolved and shown. Used when the LLM encounters this frame for the first time in a conversation.

TIER 2: CONDENSED Subsequent turns

Inherited fields omitted. Only overrides and deltas from the last Full serialization. Edges summarized by type counts unless changed.

TIER 3: SIGNAL Reference only

Node IDs + weights + active edge directions. A fingerprint. Useful when the LLM has already internalized the frame structure and only needs to know what changed.

"BTC:0.9↑ ETH:0.8→ SOL:0.6↓ | corr:2 cause:1"

Tier Negotiation Flow

Client request → frame_id + tier hint → CBP Server → Serialized payload

The client can request a specific tier, or send tier: "auto" — the server decides based on whether the conversation has previously received a Full serialization of that frame.

Wire format. The canonical wire format for all tiers is JSON Canonical Form (RFC 8785) — deterministic key ordering, normalized numbers, no whitespace outside strings. This guarantees byte-identical output across implementations, which is required for content-addressable ids (BLAKE3) and conformance testing. The textual Signal example above ("BTC:0.9↑ ETH:0.8→") is a human-readable rendering, not the wire format. Full wire-format specification: spec/wire-format.md.

V. Application Domains

1. Algorithmic Trading

high value real-time decay regime-conditional

The origin use case. A CBP frame encodes market regime, cross-asset correlations, momentum states, and Bayesian priors. The trading agent consumes a Condensed frame each tick, receiving only the delta. Conditional edges activate/deactivate based on regime classification — the agent literally sees a different market structure when conditions shift.

Node structure: frame:market → entities (assets) → states (price, vol, momentum) + priors (regime, trend). Edges: correlations and causations between macro signals and price action. Decay: event-driven on price, epoch-driven on regime.

2. Clinical Decision Support

high stakes deep inheritance regulatory

Patient context as a CBP graph. Demographics and chronic conditions are high-TTL entity nodes. Current symptoms are short-TTL state nodes. Drug interactions are conditional edges — activated only when both drugs are in the active medication list. The physician's LLM assistant sees a context-bound patient, not a raw chart dump.

Inheritance play: A hospital's standard protocol is a frame root. Every patient inherits default weightings (vitals > subjective complaints), and the physician overrides for specific cases. New residents get the institutional prior for free.

3. Legal Document Analysis

high complexity deep hierarchy slow decay

Contract clauses as entity nodes, each with state children (enforceability score, jurisdiction applicability). Cross-references between clauses as qualifies and contradicts edges. Precedent rulings as prior nodes. The LLM reviewing a contract sees not just text but the structural web of obligations and conflicts — pre-mapped.

4. DevOps / Incident Response

time-critical rapid decay cascading edges

Services as entities. Health metrics as states. Dependency chains as requires and causes edges. When service A degrades, the causes edges propagate weight boosts to downstream services — the on-call LLM agent immediately sees the blast radius, not a flat alert list. TTL is aggressive: stale metrics auto-prune within minutes.

5. Education / Adaptive Learning

personalized slow decay amplifies/inhibits

Concepts as entities. Mastery levels as states. Prerequisite chains as requires edges. Misconceptions as contradicts edges against correct concepts. The tutoring LLM receives a student frame showing exactly what they know, what's shaky, and what they can't learn yet because prerequisites aren't met.

6. Autonomous Agents (Multi-Step)

meta-layer self-referencing plan graphs

Task nodes as entities. Completion states. Dependency edges. The agent's own previous outputs as prior nodes that qualify future decisions. CBP becomes the agent's working memory — structured, prunable, and transferable between agent instances. This is where the protocol most sharply diverges from MCP: an agent using MCP calls tools; an agent using CBP understands its own situation.

Four Integration Modes

MODE 1: SYSTEM PROMPT INJECTION Simplest

The consuming app fetches a serialized frame and prepends it to the system prompt. The LLM reads context as structured text. No protocol awareness needed on the LLM side — it just reads well-organized data.

GET /v1/frame/crypto_macro?tier=full

→ Injected into system prompt as:
  <context_frame id="crypto_macro" v="1">
    ... serialized CBP payload ...
  </context_frame>
MODE 2: TOOL-MEDIATED Interactive

Register CBP as a tool the LLM can call. The model requests frames by ID when it decides it needs context. This enables lazy loading — context is pulled on demand, not pre-loaded.

tools: [{
  "name": "cbp_frame",
  "description": "Fetch structured context for a domain",
  "parameters": {
    "frame_id": "string",
    "tier": "full|condensed|signal",
    "filter_tags": ["string"]
  }
}]
MODE 3: STREAMING DELTA Real-time

WebSocket connection pushes Condensed or Signal tier updates as the underlying data changes. For trading, incident response, or any domain where context shifts faster than conversation turns.

ws://cbp-server/v1/stream/crypto_macro

→ { "delta": { "BTC.momentum": 0.68, "regime": "risk_off" },
    "edges_deactivated": ["BTC→ETH:correlates"],
    "tier": "signal" }
MODE 4: EMBEDDED SDK Agent-native

For autonomous agents, the CBP client runs in-process. The agent reads and writes to the graph — recording its own outputs as prior nodes, updating state based on actions taken. The graph becomes the agent's persistent working memory.

import { CBPClient } from '@cbp/sdk';

const ctx = new CBPClient('crypto_macro');
const frame = await ctx.resolve({ tier: 'condensed' });

// Agent takes action, records result as prior
await ctx.upsert({
  type: 'prior',
  lineage: 'entity:BTC',
  val: { action: 'long', confidence: 0.74 },
  decay: 'event',
  ttl: 7200
});

VII. Configuration Model

A CBP deployment is configured at three levels:

1. Server Configuration

{
  "server": {
    "max_nodes_per_frame": 500,
    "max_depth": 8,
    "default_decay": "epoch",
    "epoch_interval_seconds": 3600,   // GC sweep interval; advances epoch counter
    "decay_factor": 0.85,             // w multiplier per epoch for decay:epoch nodes
    "gc_policy": {
      "strategy": "prune_below_weight",
      "threshold": 0.1               // prune nodes with w below this
    },
    "edge_vocabulary": "standard_8",  // or "extended_12"
    "compression": {
      "condensed_threshold": 0.3,     // min delta ratio to trigger
      "signal_min_turns": 3           // turns before signal tier allowed
    }
  }
}
Epoch semantics (G2). The server maintains a monotonic epoch counter that advances by exactly one on each GC sweep. A node with decay: epoch has its w multiplied by decay_factor on each advance. At default settings (factor 0.85, threshold 0.1), a node starting at w=1.0 is pruned after ~14 epochs (~14 hours). Wall clock is the timer source for the sweep interval; the counter — not wall clock — is the semantic time. This makes deterministic replay possible. The formal schema is at spec/schemas/config.schema.json.

2. Frame Configuration

{
  "frame": {
    "id": "crypto_macro",
    "domain_tags": ["crypto", "macro", "trading"],
    "root_weight": 1.0,
    "root_decay": "epoch",
    "refresh_policy": "on_demand | interval:300 | event:price_update",
    "max_token_budget": 400,  // hard cap for serialized output
    "inheritance_mode": "prototypal",  // or "override_only"
    "conditional_edge_eval": "eager",  // or "lazy"
    "tokenizer": "o200k_base",        // configurable per frame (G4)
    "acl_tags": ["acl:desk_alpha"]    // bearer-token ACL (G8 v0.x)
  }
}

The tokenizer field selects the token-counting strategy for budget enforcement. Implementations register tokenizers by string identifier; the default o200k_base is a reference, not a mandate. The acl_tags field controls per-frame access: a request's bearer token must match at least one ACL tag to access the frame; an empty array means accessible to all authenticated tokens. The formal schema is at spec/schemas/frame.schema.json.

3. Client Configuration

{
  "client": {
    "preferred_tier": "auto",
    "context_window_budget": 2000,   // tokens reserved for CBP
    "multi_frame_strategy": "merge | priority_stack | isolate",
    "stale_frame_policy": "warn | refresh | use_cached",
    "write_access": true              // can the client upsert nodes?
  }
}
Token budget enforcement: The max_token_budget on a frame is a hard cap. If the serialized frame exceeds it, the serializer prunes lowest-weight nodes first, then collapses inherited fields, then falls back to a lower tier. The LLM always gets a payload that fits.

Authentication (G8 v0.x). All endpoints except /healthz require a Bearer token in the Authorization header. Per-frame access is controlled via acl_tags in the frame configuration — a request's token must match at least one ACL tag to access a restricted frame. Frames with no ACL tags are accessible to all authenticated tokens. mTLS and federation key models are deferred to v1.0. See spec/schemas/frame.schema.json.
Concurrency (G7, v0.8). Writes split into four verbs by intent. POST /v1/node is insert-only — it returns 409 Conflict with {current_v, hint} if a node with the same id already exists, and 422 Unprocessable Entity if the node's lineage chain does not terminate at any configured frame (closes the silent-orphan and cross-frame ACL-bypass vectors). PUT /v1/node/{id} is the idempotent upsert path, intended for memory-style writes with client-derived deterministic ids: body is a CbpNode with v and prev optional (v0.8.1 — server computes both and ignores any client-supplied values); on insert the server returns 201 Created with v=1; on update the server returns 200 OK with v=existing.v+1 and prev=existing.id. PATCH /v1/node/{id} is the CAS update path for concurrent-writer scenarios: body is {expectedV: number, update: {...}}; if expectedV matches, the upsert proceeds; if not, 409 Conflict with {expected_v, current_v} and the client re-reads and retries. Edges mirror the same verb set (POST /v1/edge strict-insert with src-frame ACL enforcement, PUT /v1/edge/{id} idempotent upsert). Use POST /v1/frame/{id}/import for bulk idempotent restore. Because the graph is append-mostly (invariant #6), conflicts should be rare. See spec/schemas/node.schema.json.
Persistence (v0.5). Servers configure durability under server.persistence. The memory driver (default) keeps the graph in-process only. The filesystem driver snapshots the graph to an absolute path using a debounced atomic write (1s after the last mutation, via tmp-file + rename). On start, the server hydrates from the snapshot; a malformed snapshot fails startup rather than silently proceeding. Hard-crash data-loss window equals the debounce interval. Background flush failures are delivered to an optional onPersistenceError callback on CbpServerConfig / CbpClientConfig so deployments can wire alerting. GET /v1/frame/:id/export returns the stored (non-resolved) shape — no resolver-computed fields such as edge.active — and its nodes and edges arrays must round-trip byte-identically under canonicalize() through an import/re-export cycle. POST /v1/frame/:id/import accepts the same shape, validates in three passes (schema, lineage/reference integrity, and frame-root containment — every imported node's lineage chain must terminate at the frame being imported into, and every imported edge's src must be in that frame), and either inserts every item or inserts none (422 with a structured error array). The frame-root containment pass closes the ACL-bypass vector where a caller authenticated for one frame could plant nodes that claimed lineage into another. See spec/schemas/export.schema.json and conformance vector trading/003-export-roundtrip.
Errors (v0.5). When a serialization can't fit even at Signal tier, the server returns 413 Payload Too Large with body { reason: "budget_exceeded", estimates: {full, condensed, signal}, max_token_budget, suggestion } — making invariant #1 (token budget is law) legible to clients instead of surfacing as a generic 500. POST /v1/frame/:id/import returns 422 Unprocessable Entity with { nodes_accepted: 0, edges_accepted: 0, errors: [{index, kind, issues}] } when any node or edge fails schema or reference validation; nothing is inserted in that case.
Streaming (Mode 3, v0.7). Subscribers upgrade an HTTP GET to a WebSocket at /v1/stream/{frame_id}. The upgrade request is authenticated by the same bearer-token hook as the REST surface and ACL-checked against the target frame. While subscribed, clients receive a JSON envelope on every mutation to the frame: {event: "node_upserted", frame_id, node_id, v}, {event: "node_removed", frame_id, node_id}, {event: "edge_upserted", frame_id, edge_id, v}, {event: "edge_removed", frame_id, edge_id}, or {event: "import_committed", frame_id, nodes, edges}. These are deliberately minimal "something changed" signals rather than full resolved frames — subscribers decide when to re-read via GET /v1/frame/:id/export or GET /v1/frame/:id. The full tier-negotiated push path via StreamManager.notify remains available to library consumers for richer streaming use cases.
Tier-reason observability (v0.7). GET /v1/frame/:id responses include an X-CBP-Tier-Reason header that exposes the router's decision for the delivered tier. Values: client_requested (the caller passed a concrete tier=), first_turn (auto, no prior Full in this conversation), condensed_default (auto, not yet past the signal threshold), signal_threshold_met (auto, at least signal_min_turns of Condensed have been delivered since the last Full). Closes the consumer's observability gap for debugging "why did turn N return tier X".

Query Language (CBQ)

A minimal query syntax for frame filtering — designed to be readable by both humans and LLMs:

# Fetch crypto frame, only entities with weight > 0.5
GET /v1/frame/crypto_macro?cbq=w>0.5

# Only nodes tagged with "regime"
GET /v1/frame/crypto_macro?cbq=tag:regime

# Subgraph rooted at BTC, depth 2
GET /v1/frame/crypto_macro?cbq=root:BTC,depth:2

# Active edges only (conditionals evaluated)
GET /v1/frame/crypto_macro?cbq=edges:active

# Combined: BTC subgraph, active edges, condensed tier
GET /v1/frame/crypto_macro?tier=condensed&cbq=root:BTC,depth:2,edges:active

The tier parameter is a top-level URL parameter, not part of CBQ syntax. CBQ uses the comma as its predicate separator; predicates within a query are combined with logical AND. The formal grammar is at spec/cbq.ebnf (ISO/IEC 14977 EBNF). v0.6: the CBQ predicate is applied server-side in GET /v1/frame/{id} before tier negotiation. Invalid syntax returns 400 Bad Request with a parse-error detail. When the query is applied, the server emits the response header X-CBP-CBQ-Applied: true.

X. Design Invariants

Non-negotiable properties of any conforming CBP implementation:

1. TOKEN BUDGET IS LAW

A frame serialization must never exceed its configured token budget. If the graph is too large, the serializer prunes, compresses, or drops tier — never overflows.

2. INHERITANCE IS DEFAULT

A node with no declared field inherits from its parent. Explicit is override. This is not optional — it's what makes the protocol compact.

3. EDGES ARE CONDITIONAL-FIRST

Every edge can have a conditional. Unconditional edges are syntactic sugar for conditional: always. This means the graph is always state-aware — there's no "static" mode.

4. DECAY IS MANDATORY

Every node must declare a decay policy (or inherit one). There is no immortal data in CBP. Even "permanent" entities use decay: none explicitly — which is auditable and overridable.

5. FRAMES ARE THE UNIT OF DELIVERY

You never serialize a raw node. You serialize a frame that contains nodes. This guarantees every payload has a root, a scope, and a coherent inheritance chain.

6. THE GRAPH IS APPEND-MOSTLY

Nodes are upserted, not edited in place. History is preserved via supersedes edges. This makes the graph auditable and replayable — critical for trading and compliance domains.
